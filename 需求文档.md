# 任务：开发 llm-wiki-kit —— LLM-compiled Knowledge Base Framework
你是本项目的工程实现 agent。请基于以下需求，从零创建一个可开源的 GitHub 项目。
项目名称暂定：`llm-wiki-kit`
## 0. 项目定位
`llm-wiki-kit` 是一个用于构建 “LLM 编译型知识库” 的开源框架。
它不是普通笔记模板，也不是普通 RAG 系统。它的核心目标是：
> 把不可变的原始资料 raw sources，通过 LLM agent 持续编译成可追溯、可审计、可供 ChatGPT / Claude / Gemini / Hermes / Codex 使用的 Markdown wiki。
核心思想：
- `raw/` 是不可变事实源；
- `wiki/` 是 AI 编译后的知识层；
- `schema/` 是约束 AI 如何维护知识库的规则；
- `current_draft/` 是 AI 生成的当前状态草稿；
- `current/` 是人工确认后的当前状态；
- `export_for_ai/` 是给 ChatGPT、Claude、Gemini、移动端、语音模式读取的精简知识层；
- 所有重要结论必须可追溯到 raw source；
- AI 可以整理、归类、摘要、建立链接、发现冲突，但不能直接篡改事实源，也不能未经确认修改正式 current。
## 1. 目标用户
第一类用户：个人知识库用户
- 使用 Obsidian / Markdown 管理个人资料；
- 有大量会议纪要、项目文档、周报、网页剪藏、聊天记录；
- 希望 AI 帮忙整理，而不是自己维护大量标签和状态字段。
第二类用户：产品经理 / 项目经理 / 研究人员
- 同时推进多个项目；
- 项目之间相互有交集；
- 需要区分历史材料、当前状态、待确认事项、已废弃方案；
- 需要给 ChatGPT / Claude / Gemini 提供稳定、精简、可信的上下文。
第三类用户：agent / automation 用户
- 想把框架部署到 Hermes、Codex、Claude Code、OpenClaw 或其他 agent runtime；
- 希望 agent 能通过 skills / scripts 自动执行 ingest、lint、export、mini-kb 生成等任务。
## 2. 核心问题
本项目解决以下问题：
1. raw documents 越积越多，AI 每次直接读取原始资料容易混旧、混项目、混状态；
2. 普通文件夹是树状结构，但真实项目知识是网状结构；
3. 同一份会议纪要可能同时影响多个项目、能力、概念和当前状态；
4. 项目状态会随时间变化，需要显式区分：
   - 曾经讨论过；
   - 当前已确认；
   - 正在推进；
   - 待确认；
   - 已废弃；
   - 已被替代；
5. 人不想手动维护大量 YAML、标签、反链、摘要、索引；
6. 不同 AI 工具的读取能力不同，需要一个统一的 `export_for_ai/` 消费层；
7. AI 生成内容需要审计、回滚、来源引用和人工确认机制。
## 3. 非目标
v0.1 不做以下事情：
- 不做完整 Web UI；
- 不做向量数据库；
- 不做多用户权限系统；
- 不做 Obsidian 插件；
- 不做 Google Drive 自动同步；
- 不做 Telegram / WhatsApp bot；
- 不自动调用外部 LLM API；
- 不强依赖 Hermes；
- 不强依赖 Codex；
- 不直接替代 Obsidian / Notion / Logseq；
- 不让 AI 无审核地修改正式 current。
v0.1 的重点是：
- 目录规范；
- 模板；
- CLI；
- prompts；
- source card；
- source manifest；
- lint；
- export；
- mini-kb；
- Hermes adapter skeleton；
- 示例项目。
## 4. 技术栈要求
优先使用 Python 实现 CLI。
建议技术栈：
- Python 3.11+
- `typer`：CLI 框架
- `pydantic`：配置和数据结构校验
- `rich`：终端输出
- `pyyaml`：读取 YAML / frontmatter
- `pytest`：测试
- `ruff`：lint / format
- `mypy`：可选
- `pathlib`：文件路径处理
- `tomllib` / `tomli`：读取配置
- 项目打包使用 `pyproject.toml`
请使用清晰、可维护、易扩展的结构。不要写成一个巨大脚本。

7. 核心概念定义

7.1 human/

用户本人写作区。

规则：

* AI 不得主动改写 human/；
* 除非用户明确要求，否则 agent 只读不写；
* human/ 用于用户的草稿、私人思考、临时笔记；
* 不参与默认 export；
* 不作为 AI 当前口径的事实源，除非用户明确标记。

7.2 ai_kb/raw/

不可变事实源。

规则：

* AI 不允许修改、删除、重命名 raw 文件；
* raw 文件可以是会议纪要、周报、项目文档、聊天记录、网页剪藏、数据说明；
* raw 是 source of truth；
* wiki 可以重建，raw 不能被污染。

7.3 ai_kb/wiki/

AI 编译层。

包含：

* projects/：项目页；
* domains/：业务场景页；
* capabilities/：跨项目能力页；
* concepts/：关键概念页；
* analysis/：高价值问答、边界分析、复盘；
* current_draft/：AI 生成的当前状态草稿；
* current/：人工确认后的当前状态；
* source_cards/：每份 raw 的结构化摘要卡；
* source_manifest.md：所有 raw 的索引表；
* index.md：wiki 导航；
* log.md：操作日志。

7.4 ai_kb/schema/

agent 规则层。

至少包含：

* AGENTS.md

用于告诉 Codex / Hermes / Claude Code / 其他 agent 如何维护该知识库。

7.5 ai_kb/export_for_ai/

给其他 AI 消费的精简层。

目标用户：

* ChatGPT App；
* Claude.ai；
* Gemini；
* 手机端；
* 语音模式；
* 临时会议 / 周报 / 汇报场景。

包含：

* current/：项目当前状态精选；
* mini_kb/：面向某个临时任务的压缩知识包；
* recent/：最近变化摘要。

8. CLI 功能需求

请实现 CLI 命令：llm-wiki

8.1 init

命令：

llm-wiki init ./SimonKnowledgeBase

功能：

* 创建标准目录；
* 复制模板；
* 生成根目录 README.md；
* 生成根目录 AGENTS.md；
* 生成 ai_kb/schema/AGENTS.md；
* 生成 wiki/index.md；
* 生成 wiki/log.md；
* 生成 wiki/source_manifest.md；
* 生成 portfolio/projects.md；
* 生成 portfolio/capabilities.md；
* 生成 portfolio/source_trust.md；
* 如果目标目录已有文件，必须提示并安全处理；
* 支持 --force 覆盖；
* 支持 --dry-run 预览。

验收：

* 测试 test_init.py 应覆盖：
    * 空目录初始化；
    * 已存在目录初始化；
    * dry-run；
    * force；
    * 目录结构完整性。

8.2 manifest scan

命令：

llm-wiki manifest scan ./SimonKnowledgeBase

功能：

* 扫描 ai_kb/raw/ 下所有 markdown 文件；
* 从文件名提取日期；
* 从 frontmatter 提取 metadata；
* 更新 ai_kb/wiki/source_manifest.md；
* 标记：
    * source path；
    * date；
    * type；
    * project；
    * ingest status；
    * source card status；
    * last updated；
* 不调用 LLM；
* 不修改 raw。

支持：

llm-wiki manifest scan ./SimonKnowledgeBase --dry-run
llm-wiki manifest scan ./SimonKnowledgeBase --format markdown
llm-wiki manifest scan ./SimonKnowledgeBase --format json

8.3 source-card create

命令：

llm-wiki source-card create ./SimonKnowledgeBase ai_kb/raw/meetings/2026-04-21_example.md

功能：

* 基于 raw 文件路径创建 source card 模板；
* 不自动总结内容；
* 生成可供 agent 填写的结构化文件；
* source card 存放在 ai_kb/wiki/source_cards/；
* source card 文件名应与 raw 文件对应。

source card 模板字段：

---
source_path:
source_date:
source_type:
direct_projects: []
indirect_projects: []
domains: []
capabilities: []
concepts: []
ingest_status: draft
current_impact: unknown
created_at:
updated_at:
---

正文结构：

# Source Card
## Source
## One-line Summary
## Detailed Summary
## Confirmed Facts
## Open Questions
## Decisions
## Cross-project Impact
### Direct Projects
### Indirect Projects
### Shared Capabilities
### Concepts
## Current State Impact
## Conflicts / Risks
## Pages to Update
## Notes for Human Review

8.4 prompt ingest

命令：

llm-wiki prompt ingest ./SimonKnowledgeBase ai_kb/raw/meetings/2026-04-21_example.md

功能：

* 不调用 LLM；
* 根据模板生成一段完整 ingest prompt；
* prompt 中必须包含：
    * raw 文件路径；
    * AGENTS.md 路径；
    * source_manifest 路径；
    * source_card 目标路径；
    * 要求不修改 raw；
    * 要求只更新 current_draft，不更新 current；
    * 要求输出变更摘要和需要人工确认的问题；
* 输出到 stdout；
* 支持 --copy 可选；
* 支持 --output prompt.md 保存到文件。

8.5 lint

命令：

llm-wiki lint ./SimonKnowledgeBase

功能：

不调用 LLM，只做确定性检查：

* raw 是否存在未登记到 source_manifest 的文件；
* source_manifest 中登记的文件是否不存在；
* source card 是否缺失；
* current_draft 是否有对应 current；
* current 是否超过指定天数未更新；
* export_for_ai 是否比 current 旧；
* wiki 页面是否缺少 frontmatter；
* raw 是否被误放进 wiki；
* 是否存在空目录；
* 是否存在明显的命名不规范文件。

输出：

* rich table；
* error / warning / info 分级；
* exit code：
    * 0：无 error；
    * 1：有 error；
    * 2：配置错误。

支持：

llm-wiki lint ./SimonKnowledgeBase --json
llm-wiki lint ./SimonKnowledgeBase --max-current-age 14

8.6 prompt lint-ai

命令：

llm-wiki prompt lint-ai ./SimonKnowledgeBase

功能：

* 生成给 Codex / Hermes 的 AI Lint prompt；
* 要求 agent 检查语义问题：
    * 项目状态冲突；
    * current 和 current_draft 差异；
    * 项目关系矩阵与项目页冲突；
    * 某能力被多项目引用但没有 capability 页；
    * 概念多次出现但没有 concept 页；
    * current 是否有无来源结论；
    * export_for_ai 是否过时；
    * mini_kb 是否过期。

8.7 export current

命令：

llm-wiki export current ./SimonKnowledgeBase

功能：

* 将 ai_kb/wiki/current/ 下的正式当前状态复制或聚合到 ai_kb/export_for_ai/current/；
* 生成 ai_kb/export_for_ai/current/README.md；
* 可选生成合并版：

llm-wiki export current ./SimonKnowledgeBase --single-file current_all.md

规则：

* export 只读取 current/；
* 不读取 current_draft/；
* 不读取 raw；
* 保留来源引用；
* 添加导出时间。

8.8 mini-kb create

命令：

llm-wiki mini-kb create ./SimonKnowledgeBase --topic "随心团" --purpose "评审前准备"

* 不调用 LLM；

* 存放在 ai_kb/export_for_ai/mini_kb/；
* prompt 中提示 agent 应该引用哪些页面：

10. 知识库内 schema/AGENTS.md 模板

请在 templates/schema/AGENTS.md 中写入完整规则，供用户初始化知识库后使用。

必须包含：

# LLM Wiki Maintenance Rules
## Directory Roles
- human/
- ai_kb/raw/
- ai_kb/wiki/
- ai_kb/wiki/source_cards/
- ai_kb/wiki/current_draft/
- ai_kb/wiki/current/
- ai_kb/schema/
- ai_kb/export_for_ai/
## Immutable Raw Rule
AI must never edit, delete, rename, summarize-in-place, or move files under `ai_kb/raw/`.
## Current State Rule
AI may update `current_draft/`.
AI must not update `current/` unless the user explicitly confirms.
## Source Citation Rule
Every important claim in `current/` and `current_draft/` must cite at least one source file path.
## Relationship Types
- part_of
- overlaps_with
- depends_on
- informs
- conflicts_with
- supersedes
- shares_capability
- future_extension
## Ingest Workflow
1. Read the raw source.
2. Identify date, type, direct projects, indirect projects, capabilities, concepts.
3. Create or update source card.
4. Update source_manifest.
5. Update related wiki pages.
6. Update current_draft only if current state is affected.
7. Update index and log.
8. Update export_for_ai if appropriate.
9. Output changes and human review questions.
## Query Workflow
1. Identify query type.
2. Read index and relevant current pages first.
3. Read project / capability / concept pages as needed.
4. Read source cards before raw.
5. Read raw only when necessary.
6. State the effective date.
7. Do not merge conflicting materials into a fake certainty.
## Lint Workflow
Check stale current pages, unreviewed current_draft, missing source cards, missing citations, conflicting status, orphan pages, outdated export files.

11.1 ingest.md

用途：给 Codex / Hermes / Claude Code 处理单个 raw source。

必须包含占位符：

* {{kb_root}}
* {{source_path}}
* {{agents_path}}
* {{source_manifest_path}}
* {{source_card_path}}

核心内容：

You are maintaining an LLM-compiled Markdown knowledge base.
Read and obey:
- {{agents_path}}
Task:
Ingest this raw source:
- {{source_path}}
Rules:
1. Do not modify, move, rename, or delete anything under raw/.
2. Create or update the source card.
3. Update source_manifest.
4. Identify direct projects, indirect projects, domains, capabilities, concepts.
5. Update related wiki pages.
6. If current state is affected, update current_draft only.
7. Do not update current unless explicitly instructed.
8. Update index and log.
9. Update export_for_ai if appropriate.
10. Every key claim must cite source file path.
11. Output:
   - files changed;
   - new pages created;
   - current_draft changes;
   - conflicts found;
   - questions for human review.

11.2 lint.md

语义 lint prompt。

11.3 export.md

生成 export_for_ai prompt。

11.4 mini_kb.md

生成 mini-kb prompt。

11.5 query_writeback.md

当用户问了一个高价值问题，要求 agent 将结论回写到 wiki/analysis/。

11.6 confirm_current.md

用户确认 current_draft 后，将其合并到 current 的 prompt。

hermes/

│  │  ├─ SKILL.md
│  │  └─ prompt.md
│  ├─ lint_knowledge_base/
│  │  ├─ SKILL.md
│  │  └─ prompt.md

│  │  ├─ SKILL.md

│  ├─ export_for_ai/
│  │  ├─ SKILL.md
│  │  └─ prompt.md
│  └─ confirm_current/
│     ├─ SKILL.md
│     └─ prompt.md
└─ install_skills.sh

每个 SKILL.md 应包含：

* skill name；
* purpose；
* when to use；
* inputs；
* steps；
* safety rules；
* outputs；
* files it may edit；
* files it must not edit。

默认安全规则：

* skill 不得改 raw；
* skill 不得改 current，除非 confirm_current；
* skill 默认写 current_draft；
* skill 必须输出人审问题；
* skill 必须记录 log；
* skill 必须保留来源路径。

16. 代码风格

* 函数短小；
* 类型标注；
* 错误信息清晰；
* 不吞异常；
* 不做隐式 destructive 操作；
* 所有写文件操作支持 dry-run 或至少可预测；
* 重要路径使用 pathlib.Path；
* 不把中文业务示例写死在 core；
* 示例可以有中文，但 core 要支持任意语言。

17. 安全和审计规则

必须在文档和模板中强调：

* raw is immutable；
* current requires human confirmation；
* current_draft is AI-generated；
* source paths are required；
* Git is recommended；
* export_for_ai is a consumption layer, not source of truth；
* mini-kb may expire；
* users should review diffs before committing。
